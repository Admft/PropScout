--- AI ORCHESTRATION ---
  [BUILT] Grounding stage — fetch all provider data + run calculators before LLM
  [BUILT] Tool-calling loop (max 8 turns) — LLM calls tools, then submits narrative
  [BUILT] Hosted LLM support — Anthropic Claude and/or OpenAI GPT (auto-detected)
  [BUILT] System prompts with citation rules + fair-housing constraints
  [BUILT] Structured JSON report output with inline source citations
  [BUILT] Q&A path reuses report grounding + calculator tools

2. APIs CURRENTLY IN USE
--------------------------------------------------------------------------------

DATA PROVIDERS
  RentCast API          https://api.rentcast.io
                        — /v1/properties, /v1/avm/value, /v1/avm/rent/long-term
                        — REQUIRED key: RENTCAST_API_KEY

  US Census ACS5        https://api.census.gov
                        — tract income / population
                        — OPTIONAL key: CENSUS_API_KEY (works keyless at low volume)

  FCC Census Block      https://geo.fcc.gov
                        — lat/lon → census tract lookup (no key)

  FRED                  https://api.stlouisfed.org
                        — MORTGAGE30US (30yr mortgage rate)
                        — OPTIONAL key: FRED_API_KEY

  FEMA NFHL             https://hazards.fema.gov (ArcGIS)
                        — flood zone by point query (no key)

LLM PROVIDERS (set at least one)
  Anthropic             Messages API — ANTHROPIC_API_KEY
  OpenAI                Chat Completions — OPENAI_API_KEY

INFRA (local / compose)
  Redis                 cache for provider responses + reports
  Postgres              report persistence (PostGIS + pgvector ready)

NOT USED BY CURRENT HOUSEFAX APP (legacy only)
  Apify (Zillow scraper)
  Azure Cosmos DB
  ATTOM (mentioned in master plan as alternative to RentCast — not integrated yet)

4. WHAT'S NEXT (MAPPED TO MASTER PLAN)
--------------------------------------------------------------------------------

PRIORITY A — Finish the "Essential" tier gaps (build next)
  1. Expand eval dataset to 30–50 cases (math, hallucination, fair-housing, comps)
  2. Wire PostGIS for real geospatial comps ("within 1 mile") instead of
     client-side distance filtering only
  3. Real PDF generation (server-side) + optional S3 storage
  4. Persist properties + comps into Postgres (not just reports)
  5. ATTOM as optional/fallback data provider alongside RentCast (if needed)
  6. Deploy to AWS: RDS (Postgres) + ECS/EC2 (API) + S3 (PDFs/docs)
  7. Demo polish: README positioning, architecture diagram, short demo video

PRIORITY B — Document intelligence module (distinct from core pipeline)
  8. LlamaIndex multi-document RAG over uploaded inspection reports,
     disclosures, HOA docs — with page citations
  9. Use pgvector doc_chunks table for embeddings
  10. Surface "Document findings" section in the report when docs are uploaded
  NOTE: Keep core structured-data pipeline as hand-rolled tool-calling
        (do NOT replace it with framework-driven RAG)

PRIORITY C — Product hardening
  11. Auth / user accounts (if multi-user)
  12. Rate limiting + API key management for production
  13. Report history / saved reports UI
  14. Stronger risk coverage (wildfire, insurance cost signals)
  15. Streaming report generation (optional UX improvement)

DOWN THE LINE (only once usage justifies it)
  — Terraform (multi-env)
  — Kubernetes / Helm
  — Microservices split
  — Kafka
  — Fine-tuning / LoRA
  — Standalone vector DB (Qdrant/Pinecone) if pgvector hits limits
  — Data warehouse if Postgres analytics slows
  — Multi-agent workflows (buyer / investor / realtor)

NEVER (wrong tool at this product's scale)
  — gRPC, Flink streaming
  — Framework-driven RAG for the core structured pipeline
  — Self-hosted / quantized inference in production for $500k-decision reports
  — Training a custom model from scratch
